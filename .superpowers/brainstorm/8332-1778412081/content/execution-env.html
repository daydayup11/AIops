Python 脚本在哪里执行？

Planner 生成的分析脚本需要一个执行环境，这决定了整体架构

A

后端进程内执行（subprocess / exec）

脚本由 FastAPI 后端直接用 subprocess 或 exec() 运行，生成图片文件后通过 WebSocket 把路径/base64 推给前端。

优点

  • 无需新组件，架构最简
  • 可直接访问 ClickHouse 连接池
  • 延迟低

缺点

  • exec() 有安全风险（LLM 生成代码）
  • subprocess 有沙箱但需要额外隔离
B

后端进程内执行 + 沙箱（推荐）

脚本用 subprocess 在独立进程中运行，通过环境变量传入 ClickHouse 连接参数，输出图片到 data/renders/。后端通过 HTTP 或文件路径将图片 base64 推给前端。

优点

  • 进程隔离，崩溃不影响主服务
  • 超时控制容易实现
  • 无需额外服务
  • 依然简单，改动量最小

缺点

  • LLM 生成脚本仍可做危险操作（需 prompt 约束）
  • 每次查询都需要新建 CH 连接
C

Jupyter / 独立 Python 服务

单独起一个 Python kernel 服务（如 Jupyter kernel 或自定义 gRPC 服务），复用 session 状态，脚本在 kernel 中执行并返回图片。

优点

  • 可复用数据（DataFrame 缓存）
  • 更强的沙箱（可用 seccomp/nsjail）

缺点

  • 新增一个需要维护的服务
  • 复杂度大幅增加
  • 对当前场景过度设计
